## 1. 도어 분배 규칙 (v28)

### 1.1 상수
| 상수 | 값 | 설명 |
|------|-----|------|
| DOOR_TARGET_WIDTH | 450mm | 목표 도어 너비 |
| DOOR_MAX_WIDTH | 600mm | 최대 도어 너비 |
| DOOR_MIN_WIDTH | 350mm | 최소 도어 너비 |
| MIN_REMAINDER | 5mm | 최소 잔여 (시공 공차) |
| MAX_REMAINDER | 10mm | 최대 잔여 허용 |

### 1.2 도어 너비 선택 우선순위
```
1순위: 도어 수 최소 (넓은 모듈 우선)
2순위: 목표 450mm에 가까운 너비
3순위: 잔여 공간이 작은 것
```

### 1.3 잔여 공간 처리 (v33 업데이트)
```
1. 잔여 gap을 모든 모듈에 균등 분배 (Math.floor(gap / 모듈수))
2. 균등 분배 후 나머지:
   - ≤10mm → 몰딩-모듈 사이 시공 공차로 허용
   - >10mm → 마지막 모듈에 흡수
```
- 변경 전: 잔여 전체를 마지막 모듈에만 흡수 (불균형)
- 변경 후: 모든 모듈에 균등 분배 (균형 유지)

### 1.4 모듈 패턴 (2D/1D)
```
doorCount / 2 = 몫(2D 모듈) + 나머지(1D 모듈)
2D 모듈 너비 = doorWidth × 2
1D 모듈 너비 = doorWidth
```

### 1.5 상부장 대칭 배치
```
도어 3개 이상: 1D(좌 가장자리) + 2D(중앙) + 1D(우 가장자리)
예: 
1. 5도어 → 1D + 2D + 1D + 1D OR 1D + 2D + 2D
2. 6도어 → 1D + 2D + 2D + 1D
```

### 1.6 하부장 표준 배치
```
2D 먼저 + 1D 나중 (대칭 없음)
예: 5도어 → 2D + 2D + 1D
```

### 1.7 Dead Zone (601~689mm)
- 갭이 601~689mm → 도어 1개에는 너무 크고 2개에는 너무 작음
- 해결: 서랍형으로 자동 전환

### 1.8 소규격 갭 (<350mm) 흡수
- 350mm 미만 갭은 모듈 생성 불가
- 인접한 큰 갭에 균등 분배하여 흡수

## 10. 냉장고장 규칙

### 10.1 높이 상수
| 항목 | 값 |
|------|-----|
| 상부장 최대 높이 | 400mm |
| 좌대 높이 | 60mm |
| 냉장고 상단 간격 | 15mm (고정) |
| 상몰딩 기본 | 50mm |
| 모듈 기본 깊이 | 550mm |
| 설치 필요 깊이 | 700mm |
| EL장/키큰장 기본 폭 | 600mm |

### 10.2 높이 자동 계산 (55:45 비율)
```
상부장H = min(400, 전체H - 냉장고H - 15 - 상몰딩H)
모듈바디H = 전체H - 상몰딩H - 상부장H - 좌대H
중단H = floor(모듈바디H × 0.55)
하단H = floor(모듈바디H - 중단H)
```

### 10.3 브랜드별 여유 간격
| 브랜드/타입 | 좌우 간격 | 유닛간 간격 |
|-----------|----------|-----------|
| LG 빌트인 | 4mm | 8mm |
| LG 프리스탠딩 | 50mm | 0mm |
| 삼성 비스포크 | 12mm | 10mm |
| 삼성 인피니트 | 5mm | 10mm |
| 삼성 프리스탠딩 | 50mm | 0mm |

### 10.4 냉장고장 너비 계산
```
냉장고W = 좌우간격×2 + 유닛합산W + (유닛수-1)×유닛간간격
```

## 14. 손잡이 규칙 (v33 신규)

- **기본**: handleless (매립형) — 모든 카테고리 적용
- **하부장**: 도어 뒤로 손을 넣어서 열 수 있는 형태

- **금지**: Chrome bar handle, push-to-open, 외부 노출 손잡이
- AI 이미지 생성 프롬프트에도 handleless 강제 적용

## 15. AI 이미지 생성 아키텍처 (v33 신규)

### 15.1 모델
- **Gemini 3.1 Flash Image Preview** (`gemini-3.1-flash-image-preview`)
- 벽분석 + 이미지 생성 모두 동일 모델 사용

### 15.2 파이프라인 (n8n 제거, MCP 서버 직접)
```
프론트엔드 → POST /api/generate → MCP 서버
  → Step 1: Gemini 벽분석 (배관/환풍구 위치)
  → Step 2: 프롬프트 생성 (레이아웃 + 스타일 + handleless)
  → Step 3: Gemini 닫힌문 이미지 생성
  → Step 4: Gemini 열린문 이미지 생성
  → 응답: { background, closed, open }
```

### 15.3 프롬프트 제약
- n8n Cloud 제한 없음 (직접 API 호출로 1000자+ 가능)
- handleless 3중 강조 (CRITICAL + STRICT + furniture door 표현)
- kitchen_layout별 layoutDesc 자동 적용

## 16. 3D 뷰 (v33 신규)

### 16.1 기술
- Three.js (CDN, WebGL)
- OrthographicCamera (2D 스타일) + 회전 시 PerspectiveCamera 자동 전환

### 16.2 뷰 프리셋
| 버튼 | 카메라 위치 | 설명 |
|------|-----------|------|
| Front | 정면 | 정면도 (기존 Front View 대체) |
| Top | 상면 | 상면도 (기존 Top View 대체) |
| Iso | 등각 | 아이소메트릭 (기존 Iso View 대체) |
| Free | 자유 | 마우스 드래그로 자유 회전 |

### 16.3 모듈 배치 규칙
- 하부장: 상판 대비 앞뒤 10mm inset (좌우 밀착)
- 상부장: 틈새 없이 밀착
- 라벨: 모듈 앞쪽(+Z) 표시, 폰트 x9 확대
- 분배기: 💧 파란색 원기둥 (벽면)
- 환풍구: 🌀 회색 박스 + 그릴 (벽면 상부)
